# WordPress + AI Development Reference Guide

**Best Practices for Pods, Gravity Forms, Gravity SMTP, Spellbook (Gravity Wiz), Gravity Connect, and GravityView**

Integrating Artificial Intelligence (AI) into the WordPress development stack has transitioned from simple content generation to complex, agentic automation, and structured model-to-application interactions. This document compiles technical architectural patterns, performance guidelines, security guardrails, and direct resource links for building with WordPress custom data structures and the Gravity ecosystem.

---

## 1. Architectural Strategy & Tool Best Practices

### Pods (Custom Content Types & Fields)

Pods provides a robust framework for managing custom post types (CPTs), taxonomies, and custom database tables. When interacting with AI to build or write code for Pods:

- **Declare the DB Schema to LLMs:** Always explicitly state to your AI assistant (e.g., Cursor, Claude, Copilot) whether your Pod is using standard WordPress `wp_postmeta` (custom fields) or Advanced Content Types (ACTs). ACTs bypass the standard postmeta structure to write directly to custom database tables. An LLM needs this distinction to write correct, high-performance SQL/WP_Query operations.
- **Relationship Mapping:** Pods allows bi-directional relationships. When using AI agents to dynamically seed or sync fields, ensure the model generates and targets raw arrays of target IDs rather than text titles, maintaining relational integrity.
- **WordPress Abilities API Integration:** Expose Pod fields to core-level REST API hooks to allow external LLM workflows (WordPress/ai) to read/write structure natively.

### Gravity Forms

Gravity Forms is a highly extensible transactional endpoint. Modern systems treat forms as structured input nodes for AI routing.

- **Turn Forms into JSON Schemas:** Use tools or custom filters to serialize form structures into Model Context Protocol (MCP) compatible files (WebMCP). This allows autonomous AI web agents to interact with, populate, and programmatically submit form data safely.
- **Prevent Token Draining & AI Spam:** Standard honeypots may fail against sophisticated browser-based AI agents. Integrate server-side rate limits, Cloudflare Turnstile, and programmatic field submission gates to prevent malicious bots from generating costly, spammy AI executions.

### Gravity SMTP

Dynamic systems rely on SMTP to reliably transmit notifications generated by back-end LLMs.

- **Crucial Security Patch Check (CVE-2026-4020):** Ensure that Gravity SMTP is upgraded to v2.1.5 or newer. Versions v2.1.4 and older contain an unauthenticated REST API endpoint vulnerability that leaks critical API keys (SendGrid, Amazon SES, Resend, Mailgun).
- **Credential Isolation:** Store your API credentials in environment variables (`wp-config.php`) rather than the WordPress database (`wp_options`). This ensures that even if an AI assistant or third-party visual builder has read access to the database, your mail server credentials remain secure.

### Gravity Wiz: Spellbook & Various Perks

Spellbook is the modernized control hub that optimizes, licenses, and unifies the entire Gravity Perks and Gravity Connect ecosystem, replacing legacy individual helper setups.

- **Use Spellbook to Reduce Overhead:** Transition legacy setups to the unified Spellbook bootstrap. This reduces PHP memory footprints during memory-intensive LLM/API execution loops.
- **Dynamic Seeding with GP Populate Anything:** Combine GP Populate Anything with custom query hooks. This permits your forms to pull complex post/field data based on active user actions and pass structured, filtered values straight into AI feeds.
- **Advanced Select Optimization:** For dynamic lists returned by AI models, run Gravity Forms Advanced Select (powered by Choice.js) to avoid page-load lag on extensive selections.

### Gravity Connect (OpenAI Integrations)

Gravity Connect establishes lightweight, low-latency, direct API connections to external LLM services, reducing reliance on bloated third-party integration platforms like Zapier.

- **Fields vs. Feeds Strategy:**
  - **OpenAI Fields:** Use when real-time, interactive generation is required before form submission (e.g., dynamic translation, resume grammar optimization, real-time summary).
  - **OpenAI Feeds:** Use for heavy post-submission processing (e.g., automated email drafting, database tagging, document routing). This avoids slowing down front-end rendering.
- **Endpoint Optimization via OpenRouter:** Route requests through API aggregators like OpenRouter by updating the custom base API URL. This allows you to hot-swap expensive models for cheaper open-source alternatives (such as Llama-3 or Mistral) without rewriting code.
- **Enforce Strict Token Ceilings:** Always define an explicit Maximum Tokens parameter on your Connect feeds to prevent run-away conversational loops from spiking API billing.

### GravityView

GravityView handles output rendering of form data without complex query template design.

- **MCP Natural Language Prototyping:** Utilizing Model Context Protocol (MCP) clients, you can feed a Claude Desktop application the structure of your forms, and direct the LLM to design or modify the HTML/CSS views dynamically.
- **Conditional Admin Workflows:** Use automated classification from Gravity Connect (e.g., categorizing submissions as "Flagged") and create specific, restricted GravityViews for manual admin review.

## 3. Implementation Workflow: `llms.txt` and `.cursorrules`

To ensure that AI engines like Claude, Copilot, or Cursor write flawless code targeting this stack, place these configuration files in your project directory root.

### Project `.cursorrules` Structure

Save at the root of your WordPress development repository:

```json
{
  "project_type": "wordpress_ai_gravity_stack",
  "rules": [
    "Always check if Custom Fields are stored in wp_postmeta or via Pods Advanced Content Types (ACTs) before writing DB queries.",
    "When interacting with Gravity Forms, prioritize using the formal API methods (GFAPI::get_entry, GFAPI::update_entry) over direct DB updates.",
    "Do not write legacy PHP functions for Gravity Perks; utilize updated namespaces managed via Gravity Wiz Spellbook.",
    "When configuring OpenAI integration, ensure all feeds enforce a strict token safety limit to prevent run-away API execution loops.",
    "Ensure all custom email actions utilize Gravity SMTP delivery wrappers and load external authentication parameters safely from wp-config.php, never hardcoded.",
    "Never generate raw SQL queries targeting custom tables without checking Pods metadata structure registration."
  ]
}
```

### Universal `llms.txt` Context File

Place at `https://yourdomain.com/llms.txt` to allow automated system developers, scraping tools, and MCP pipelines to understand your integration framework:

```
# WordPress AI and Gravity Integration Infrastructure
This workspace is built utilizing the Pods Framework, Gravity Forms, Gravity SMTP, Gravity Wiz (Spellbook), and Gravity Connect pipelines.

## Project Schemas & Code Bases
- Pods Framework Core: https://github.com/pods-framework/pods
- Gravity Wiz Snippet Framework: https://github.com/gravitywiz/snippet-library
- Gravity Connect OpenAI Framework: https://gravitywiz.com/gravity-forms-openai/

## Development Conventions
1. Data Integrity: Keep custom fields structured. Rely on native helper parameters like `pods_field()` or `GFAPI` class methods.
2. Security Vulnerability Preventions: Gravity SMTP installations must be kept at v2.1.5 or newer to avoid CVE-2026-4020 API token exposures.
3. API Efficiency: Maximize the use of Gravity Connect OpenAI Feeds rather than Fields for background tasks to reduce client wait times.
```
